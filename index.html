Introduction to Version Control

Understand what version control is, why it exists, and how Git and GitHub fit into the picture.

1.1 What is Version Control?

Version control is a system that records changes to a file or set of files over time so that you can recall specific versions later. Think of it like the "undo history" in Microsoft Word, but far more powerful, structured, and designed for software development at scale.

In everyday terms, imagine you are writing a novel. You start with a draft, make edits, rewrite chapters, and at some point realize the original Chapter 3 was better. Without version control, you would have to manually keep copies — "draft1.docx", "draft2_final.docx", "draft2_FINAL_v2.docx" — which quickly becomes chaotic. Version control solves exactly this problem by automatically tracking every change, who made it, when, and why.

In software development, version control is absolutely essential. A single project may contain thousands of files and be edited by dozens of developers simultaneously. Without a versioning system, keeping track of changes, avoiding conflicts, and rolling back errors would be practically impossible. Version control systems (VCS) provide a structured way to manage this complexity.

There are two main types of version control systems: Centralized Version Control Systems (CVCS) like SVN (Subversion) and Perforce, and Distributed Version Control Systems (DVCS) like Git and Mercurial. In CVCS, there is one central server that stores all version history, and every developer connects to it. In DVCS, every developer has a complete copy of the entire history on their local machine, which provides speed, redundancy, and the ability to work offline.

💡 Purpose: To track, manage, and collaborate on file changes systematically across time, teams, and projects.

Real-World Use Cases

  • 🔧 Software teams tracking code changes across releases
  • 📝 Writers collaborating on documentation or books
  • 🎨 Designers versioning UI/UX files (Figma has version history too)
  • 🏢 Large corporations managing enterprise codebases with hundreds of developers
  • 🧑‍🔬 Researchers versioning data analysis scripts

1.2 Why Version Control is Important

Version control is not just a tool — it is a fundamental best practice of professional software development. Here is why every developer, whether working alone or in a team, needs version control:

History and Accountability: Every change ever made to a project is recorded with a timestamp and the author's identity. This creates a complete audit trail. If a bug appears in a production system, developers can look back through the history, find exactly which change introduced the bug, and understand why that change was made by reading the commit message.

Collaboration Without Chaos: Multiple developers can work on the same project simultaneously. Version control systems have built-in mechanisms to merge changes from different contributors intelligently. Without this, two developers editing the same file would simply overwrite each other's work.

Safe Experimentation: Branching (which we cover in Module 5) allows developers to try new features, refactor code, or experiment with ideas in an isolated copy of the project — without risking the stable, working version. If the experiment fails, the branch is simply discarded. If it succeeds, it is merged into the main project.

Backup and Recovery: Because the full history is stored — both locally and on remote servers like GitHub — if a developer's computer crashes or files are accidentally deleted, the entire project can be recovered instantly. Version control is effectively a continuous, automated backup system.

Code Review and Quality: Modern version control workflows include code review processes, where changes are reviewed by peers before being merged. This dramatically improves code quality and helps junior developers learn from senior ones.

✅ Advantages

  • Complete history of every change
  • Easy rollback to any previous state
  • Parallel development via branches
  • Better team collaboration
  • Automatic conflict detection
  • Accountability and transparency

⚠️ Limitations

  • Learning curve for beginners
  • Complex merge conflicts in large teams
  • Storage overhead for binary files
  • Requires discipline in commit messages
  • Misuse can create messy histories

1.3 History of Git

Git was created by Linus Torvalds in 2005 — the same person who created the Linux operating system kernel. The story behind Git's creation is fascinating and instructive.

The Linux kernel project, one of the most complex open-source software projects in history, was using a proprietary version control system called BitKeeper from 2002 to 2005. BitKeeper was free for the Linux project under special terms. However, in 2005, a dispute arose between the Linux community and BitKeeper's company, and the free license was revoked. This left the Linux project without a version control system that could handle their enormous scale.

Linus Torvalds, dissatisfied with existing alternatives (CVS, SVN) which he felt were slow and poorly designed, decided to build a new system from scratch. He set out with very specific goals: it had to be extremely fast, it had to support distributed workflows (no single point of failure), it had to protect against data corruption, and it had to be free and open source. In just about two weeks of intense development, Torvalds created the first version of Git. By June 2005, Git was already managing the Linux kernel's codebase.

Git's name, according to Torvalds, is a British slang word meaning "unpleasant person" — and Torvalds jokingly said he names all his projects after himself. Over the following years, Git was developed by Junio Hamano (who has maintained it since 2005) and contributions from thousands of open-source developers.

1.4 What is Git?

Git is a free, open-source, distributed version control system designed to handle everything from small personal projects to very large projects with speed and efficiency. It is the most widely used version control system in the world today, used by virtually every major technology company and the majority of software developers globally.

Git operates primarily through your command line interface (CLI) — a text-based terminal where you type commands. While graphical user interfaces (GUIs) exist for Git, understanding the command line gives you complete control and is expected in professional environments. Git stores data in a special format — not as a list of file differences, but as a series of snapshots of your entire project at each point in time. This makes Git extremely fast and efficient.

When you initialize a Git repository (project), Git creates a hidden folder called .git at the root of your project. This folder contains the entire history of your project — every commit, every branch, every tag. This means your project's complete version history travels with it wherever you copy the folder. This is what makes Git "distributed" — you do not need a network connection to access the full history. You can commit, branch, merge, and view history entirely offline.

Git's core workflow revolves around three areas: the Working Directory (where you edit files), the Staging Area (where you prepare changes for a commit), and the Repository (where commits are permanently stored). Understanding this three-stage workflow is one of the most important foundational concepts in Git.

1.5 What is GitHub?

GitHub is a web-based platform and hosting service for Git repositories. While Git is the version control tool itself (a command-line program installed on your computer), GitHub is a cloud service that hosts your Git repositories online, making them accessible from anywhere and shareable with anyone.

Think of Git as the engine of a car and GitHub as the garage and showroom where you park, maintain, and show off your car. You can use Git without GitHub (storing your project only on your local machine), but you cannot use GitHub without Git (since GitHub is built entirely around Git's infrastructure).

GitHub provides a rich web interface that makes working with Git repositories much more visual and accessible. Beyond just hosting code, GitHub has evolved into a complete collaboration and project management platform. It offers features like Pull Requests (a formal process to propose, discuss, and review code changes), Issues (a bug tracker and task management system), GitHub Actions (automated workflows for building, testing, and deploying code), GitHub Pages (free website hosting from your repository), and GitHub Discussions (community forums within projects).

GitHub is also the world's largest platform for open-source software. Projects like Linux, React, TensorFlow, VS Code, Node.js, and millions of others are hosted on GitHub. Being active on GitHub is not just a technical skill — it is a professional portfolio that employers examine when evaluating candidates for software engineering roles.

1.7 Why Companies Use GitHub

GitHub has become the de facto standard platform for software development at companies of all sizes — from two-person startups to Fortune 500 enterprises. Understanding why companies choose GitHub helps you appreciate the scale and importance of what you are learning.

Centralized Code Repository: GitHub serves as the single source of truth for all code. Every team member, across offices and time zones, works from the same repository. This eliminates the "which version is the latest?" confusion that plagues organizations without a proper VCS.

Code Review and Quality Gates: The Pull Request (PR) workflow forces all code changes to be reviewed by at least one other developer before being merged into the main codebase. This built-in quality gate catches bugs, enforces coding standards, and facilitates knowledge transfer. Many companies configure GitHub to require a minimum number of approvals, passing automated tests, and other checks before a PR can merge.

Automation and CI/CD: GitHub Actions allows companies to automate their entire software delivery pipeline. When code is pushed, GitHub can automatically run tests, check code quality, build the application, and even deploy it to production — without any manual intervention. This is the core of modern DevOps practices.

Developer Analytics and Insights: GitHub provides detailed analytics about repository activity, contributor statistics, code frequency, and more. Teams use this data to understand development velocity, identify bottlenecks, and plan sprints.

Security: GitHub offers Dependabot (automatic security alerts and updates for vulnerable dependencies), code scanning (finding security vulnerabilities in code), and secret scanning (detecting accidentally committed API keys and passwords). These features are critical for enterprise security compliance.

🎤 Interview Questions — Module 1

Q1: What is version control and why is it important?
Version control is a system that records changes to files over time, allowing you to recall specific versions, track who made changes, collaborate without overwriting work, and recover from errors. It is important because it enables safe collaboration, maintains complete project history, supports experimentation via branches, and serves as an automated backup system.
Q2: What is the difference between Git and GitHub?
Git is a free, open-source, distributed version control system that runs locally on your computer. In a distributed VCS like Git, every developer's local machine contains a complete copy of the entire project history. On Windows, the most common and recommended way to install Git is through the official Git for Windows installer, which also installs Git Bash — a terminal emulator that provides a Unix-like command-line environment on Windows.

2.2 Creating a GitHub Account

Creating a GitHub account is free and straightforward. However, your GitHub username is a professional identifier — think of it as your developer brand name. It will appear on your contributions, in your profile URL (github.com/yourusername), and on every commit you push. Choose a username that is professional, memorable, and ideally close to your real name.

1
Visit github.com and click "Sign Up"
2
Enter your email, create a strong password, and choose a username
3
Verify your email address via the confirmation email
4
Complete the optional onboarding quiz (helps GitHub personalize recommendations)
5
Enable Two-Factor Authentication (2FA) immediately — this is critical for security
6

⚠️ Username Tips: Avoid numbers at the end (john123), avoid underscores if possible, keep it short and professional. Examples: johnsmith, jsmith-dev, j-smith.

2.3 Git Configuration

Before you can make your first commit, you need to tell Git who you are. Git associates your name and email address with every commit you make. This information is embedded permanently into the commit history and is how GitHub knows to link your commits to your account. This configuration is done using the git config command.

Git has three levels of configuration: System (applies to all users on the computer), Global (applies to all repositories for your user), and Local (applies only to the current repository). For personal setup, you almost always use --global.

Git Bash / Terminal
# Set your name (use your real name — it will be in commits)
git config --global user.name "John Smith"

# Set your email (MUST match your GitHub account email)
git config --global user.email "john@example.com"

# Set your preferred text editor (VS Code)
git config --global core.editor "code --wait"

# Set default branch name to 'main' (modern standard)
git config --global init.defaultBranch main

# Enable color output for readability
git config --global color.ui auto

# View all your current settings
git config --list

# View a specific setting
git config user.name
📌 Important: Your email in git config MUST match the email on your GitHub account. Otherwise, your commits won't be linked to your GitHub profile and won't appear in your contribution graph.

2.4 Understanding Git Bash and Terminal

Git Bash (on Windows) and the Terminal (on macOS/Linux) are your command-line interfaces — text-based environments where you type commands to interact with your operating system and Git. Unlike clicking buttons in a graphical app, the command line gives you precise, powerful control over what Git does.

Learning the terminal feels intimidating at first, but you only need a handful of commands to be productive. The terminal follows a pattern: you type a command and press Enter, and the system executes it and shows you the output. Commands take the form of: command [options] [arguments]. For example, git add . is the command git, the subcommand add, and the argument . (which means "everything in the current directory").

Essential Terminal Commands
pwd          # Print Working Directory — shows where you are
ls           # List files in current directory
ls -la       # List all files including hidden ones
cd foldername  # Change Directory into a folder
cd ..        # Go up one level
mkdir myproject  # Make a new directory/folder
touch file.txt   # Create a new empty file
clear        # Clear the terminal screen
cat file.txt # Display file contents

2.5 SSH Key Setup (Connecting Git to GitHub)

To securely communicate between your local Git and your GitHub account without entering your password every time, you set up an SSH key pair. SSH (Secure Shell) uses a pair of cryptographic keys — a private key (kept on your computer) and a public key (shared with GitHub). When you connect, GitHub verifies your identity using these keys without ever transmitting your password.

SSH Setup Steps
# Step 1: Generate SSH key pair
ssh-keygen -t ed25519 -C "your_email@example.com"
# Press Enter to accept defaults

# Step 2: Start the ssh-agent
eval "$(ssh-agent -s)"

# Step 3: Add your key to the agent
ssh-add ~/.ssh/id_ed25519

# Step 4: Copy your PUBLIC key to clipboard
cat ~/.ssh/id_ed25519.pub
# Copy the output

# Step 5: Go to GitHub → Settings → SSH and GPG keys → New SSH key
# Paste the public key and save

# Step 6: Test the connection
ssh -T git@github.com
# Should see: "Hi username! You've successfully authenticated"

🎤 Interview Questions — Module 2

Q1: What happens if your git config email doesn't match your GitHub account email?
Your commits will still be made, but they won't be linked to your GitHub profile. This means they won't appear in your contribution graph, won't be credited to your account in repository statistics, and your avatar won't show next to commits in the GitHub UI. Always ensure the emails match.
Q2: What is SSH and why is it preferred over HTTPS for GitHub?
SSH (Secure Shell) is a cryptographic protocol for secure network communication. With SSH key authentication, you never transmit your password and don't need to enter credentials with each git push/pull. HTTPS requires entering a Personal Access Token (PAT) each time or storing credentials. SSH is more secure and convenient for frequent use.

Git Fundamentals — Core Commands

Master the essential Git commands with detailed explanations, syntax, and real-world examples.

git init

3.1 git init — Initialize a Repository

The git init command is the very first command you run when starting a new project with Git. It initializes a new Git repository in your current directory by creating the hidden .git folder. This folder is the heart of your Git repository — it stores the entire version history, configuration, and all Git data. Without this folder, the directory is just a regular folder; with it, Git tracks every change.

You only need to run git init once per project, at the very beginning. If you accidentally run it in the wrong directory, you can undo it by deleting the .git folder (rm -rf .git). Think of git init as "turning on the camera" for your project — from this point forward, Git is watching and ready to record.

Syntax & Example
# Syntax
git init [directory-name]

# Example 1: Initialize Git in your CURRENT directory
mkdir my-project
cd my-project
git init
# Output: Initialized empty Git repository in /path/to/my-project/.git/

# Example 2: Create and initialize a new directory in one command
git init my-new-app
cd my-new-app

# Verify the .git folder exists
ls -la  # You'll see the .git folder
Real-World Use Case: You start a new web development project. You create a project folder, run git init, and immediately begin tracking your HTML, CSS, and JavaScript files. Every change you make can now be committed, giving you a complete history of your project's evolution.
git clone

3.2 git clone — Clone a Repository

git clone creates a complete copy of an existing remote repository on your local machine. Unlike git init (which starts fresh), git clone downloads everything — all files, all branches, and the entire commit history. It automatically sets up a connection to the original repository (called "origin"), so you can push and pull changes.

Cloning is how you "get" an existing project — whether it's your own project from GitHub, a team project from your company's repository, or an open-source project you want to contribute to. After cloning, you have a full local copy and can start working immediately.

Syntax & Example
# Syntax
git clone <repository-url> [local-directory-name]

# Clone using HTTPS
git clone https://github.com/username/repository-name.git

# Clone using SSH (recommended after SSH setup)
git clone git@github.com:username/repository-name.git

# Clone into a specific folder name
git clone https://github.com/facebook/react.git my-react-copy

# Clone a specific branch
git clone -b develop https://github.com/username/repo.git

# Clone with limited history (faster for large repos)
git clone --depth 1 https://github.com/torvalds/linux.git
git status

3.3 git status — Check Repository Status

git status is arguably the command you will use most frequently. It shows you the current state of your working directory and staging area — which files have been modified, which are staged for commit, which are new (untracked), and which have been deleted. It is your "what's going on right now?" command.

Running git status before almost every other operation is a great habit. It tells you exactly where you are and prevents mistakes like accidentally committing the wrong files or forgetting to stage changes. The output is color-coded: red for unstaged changes, green for staged changes.

Syntax & Example
# Syntax
git status

# Short format (one line per file)
git status -s
# M  = Modified and staged (green M)
# M  = Modified but not staged (red M)
# A  = New file staged
# ?? = Untracked file

# Example output after modifying index.html
# On branch main
# Changes not staged for commit:
#   (use "git add <file>..." to update what will be committed)
#         modified:   index.html
# Untracked files:
#         style.css
git add

3.4 git add — Stage Changes

git add moves changes from your working directory to the staging area (also called the "index"). Think of the staging area as a waiting room — changes sit here until you are ready to commit them. This two-step process (add → commit) gives you fine-grained control over exactly what goes into each commit. You can modify five files but choose to stage and commit only two of them, keeping the other three as works in progress.

The staging area is one of Git's most powerful and misunderstood features. It lets you craft logical, coherent commits rather than lumping all your current work into one giant change. Good practice is to make each commit represent one logical change or fix, which makes the history much easier to read and understand.

Syntax & Example
# Stage a specific file
git add index.html

# Stage multiple specific files
git add index.html style.css app.js

# Stage ALL changes in current directory (most common)
git add .

# Stage all changes in the entire repository
git add -A

# Stage parts of a file interactively (advanced)
git add -p index.html

# Unstage a file (remove from staging but keep changes)
git restore --staged index.html
git commit

3.5 git commit — Save Changes to History

git commit takes everything in your staging area and permanently saves it as a snapshot in your repository's history. A commit is like taking a photograph of your project at a specific moment. Each commit has a unique identifier (a SHA-1 hash like a3f72b8), your name and email, a timestamp, and most importantly — a commit message describing what changed and why.

Writing good commit messages is a professional skill. A good commit message follows the convention: a short summary (max 72 characters) on the first line, optionally followed by a blank line and a detailed description. The summary should complete the sentence: "If applied, this commit will..." Examples of good commit messages: "Fix: resolve null pointer exception in user login", "Feat: add dark mode toggle to settings page", "Docs: update README with installation instructions".

Syntax & Example
# Basic commit with inline message
git commit -m "Add homepage navigation menu"

# Commit with detailed message (opens text editor)
git commit

# Stage all tracked files AND commit in one command
git commit -am "Fix login validation bug"
# Note: -a only stages MODIFIED files, not new untracked files

# Amend the last commit (fix message or add forgotten files)
git commit --amend -m "Corrected commit message"
# Warning: Never amend commits already pushed to shared branches!

# Conventional commit format (industry standard)
# type(scope): description
git commit -m "feat(auth): add OAuth2 Google login"
git commit -m "fix(ui): correct button alignment on mobile"
git commit -m "docs(readme): add API documentation section"
Conventional Commits: Many companies follow the Conventional Commits specification. Common types: feat (new feature), fix (bug fix), docs (documentation), style (formatting), refactor (code restructuring), test (adding tests), chore (maintenance).
git push

3.6 git push — Upload to Remote

git push uploads your local commits to a remote repository (like GitHub). Until you push, your commits exist only on your local machine — no one else can see them, and they are not backed up to GitHub. Pushing makes your work visible to collaborators and triggers any CI/CD automation configured in the repository.

The first time you push a new branch, you need to set the "upstream" remote branch with -u origin branch-name. After that, you can simply run git push without arguments. "Origin" is the default name Git gives to the remote repository you cloned from or linked to.

Syntax & Example
# Push current branch to its upstream (after -u is set)
git push

# First-time push: set upstream and push
git push -u origin main
git push -u origin feature/login

# Push a specific branch
git push origin feature/new-ui

# Push all local branches
git push --all origin

# Force push (DANGEROUS — only for personal branches)
git push --force origin feature/my-branch
# Or safer force push (won't overwrite others' changes)
git push --force-with-lease origin feature/my-branch
git pull

3.7 git pull — Download and Merge Changes

git pull does two things in one command: it fetches changes from the remote repository and immediately merges them into your current local branch. It is the standard way to update your local copy with the latest changes your teammates have pushed. In team environments, you should run git pull at the start of every working session to ensure you are working with the most current code.

Important: git pull = git fetch + git merge. Some developers prefer to use git fetch followed by a manual review before merging, which gives more control. However, for straightforward daily updates, git pull is perfectly appropriate.

Syntax & Example
# Pull latest changes from current branch's upstream
git pull

# Pull from specific remote and branch
git pull origin main

# Pull and rebase instead of merge (cleaner history)
git pull --rebase origin main

# Pull all branches
git pull --all
git fetch

3.8 git fetch — Download Without Merging

git fetch downloads changes from the remote repository but does NOT merge them into your working files. It updates your local knowledge of what the remote looks like, without actually changing any files you are working on. This is useful when you want to see what others have pushed before deciding whether and how to integrate those changes.

After fetching, you can use git log origin/main to see what changed remotely, and then decide to merge, rebase, or take a different action. This is considered a safer, more deliberate workflow than blind-pulling.

Syntax & Example
# Fetch from all remotes
git fetch

# Fetch from specific remote
git fetch origin

# Fetch a specific branch
git fetch origin main

# After fetching, see what changed
git log origin/main --oneline

# Then merge manually if ready
git merge origin/main
git log

3.9 git log — View Commit History

git log displays the commit history of your repository in reverse chronological order (newest first). Each entry shows the commit hash, author, date, and message. The log is your project's story — every decision, fix, and feature ever added is recorded here. Knowing how to read and filter the log is essential for debugging, code reviews, and understanding a project's evolution.

The default output can be verbose. Git provides many formatting options to make the log more readable. The --oneline flag is extremely commonly used for a compact view. --graph draws a visual branch and merge diagram. Learning to use these options helps you quickly navigate even very large project histories.

Syntax & Example
# Full log (press Q to exit)
git log

# One-line format (most commonly used)
git log --oneline

# Beautiful graph view
git log --oneline --graph --all --decorate

# Limit to last N commits
git log -5 --oneline

# Log by a specific author
git log --author="John Smith"

3.10 git diff — Compare Changes

git diff shows you the exact line-by-line differences between different states of your files. It uses a special format: lines starting with + (shown in green) are additions, and lines starting with - (shown in red) are deletions. Lines with no prefix are unchanged context.

There are several different things you might want to compare: your working directory vs. the staging area, the staging area vs. the last commit, or two specific commits/branches against each other. Understanding which comparison mode to use helps you review your work before committing and understand exactly what changed between any two points in history.

Syntax & Example
# Compare working directory vs staging area (unstaged changes)
git diff

# Compare staging area vs last commit (staged changes)
git diff --staged
# (also works: git diff --cached)

# Compare two commits
git diff abc123 def456

# Compare two branches
git diff main feature/new-login

# Compare specific file across branches
git diff main..feature/ui -- index.html

# Summary of changes (just filenames)
git diff --stat main feature/new-login

Both download changes from the remote. git fetch downloads the changes but does NOT merge them into your working branch — it updates your remote-tracking branches only. git pull = git fetch + git merge. 4.1 Local vs Remote Repositories

A local repository is the copy of your project that lives on your own computer. It is the .git folder in your project directory, containing the full history and all branches. You can commit, branch, merge, and perform virtually all Git operations locally without any internet connection. The local repository is where you do all your actual development work.

You can have multiple remotes. For example, when contributing to open-source projects, you typically have two remotes: origin (your personal fork on GitHub) and upstream (the original project's repository). This pattern is covered in detail in Module 9 on Open Source Contributions.

Branching and Collaboration

Understand Git's most powerful feature — branches — and how they enable safe parallel development.

5.1 What is a Branch?

A branch in Git is an independent line of development — a lightweight, movable pointer to a specific commit. Think of your project as a river: the main river (your main branch) flows forward continuously. When a developer needs to work on a new feature, they create a "tributary" branch — a parallel stream that diverges from the main river. Work can proceed in this tributary without disturbing the main river, and when the feature is complete and tested, the tributary merges back into the main river.

Technically, a branch is just a text file in your .git/refs/heads/ folder that contains the SHA-1 hash of the latest commit on that branch. This makes branches incredibly lightweight — creating a new branch takes milliseconds and costs almost no storage. This is one of Git's most significant advantages over older version control systems where branching was expensive and slow.

Git always has a special pointer called HEAD that indicates which branch you are currently on. When you make a commit, the branch HEAD is pointing to automatically moves forward to the new commit. This is how branches "grow" forward with each commit.

5.3 Merging Branches

When feature development is complete, the branch is merged back into the main branch. Git has two main types of merge strategies:

Fast-Forward Merge: If the main branch hasn't changed since the feature branch was created, Git simply moves the main branch pointer forward to the feature branch's latest commit. No new merge commit is created. The history looks linear and clean.

Three-Way Merge: If both branches have new commits since they diverged, Git creates a new merge commit that has two parents — one from each branch. This preserves the complete history of both lines of development and is the most common type of merge in team environments.

Merge Commands
# Step 1: Switch to the branch you want to merge INTO
git switch main

# Step 2: Merge the feature branch
git merge feature/user-authentication

# Merge with a commit message (no-fast-forward)
git merge --no-ff feature/payment -m "Merge: add payment integration"

# Squash all feature commits into one commit
git merge --squash feature/cleanup
git commit -m "feat: implement user profile cleanup"

# Abort a merge in progress (if conflicts arise)
git merge --abort

5.4 Merge Conflicts

A merge conflict occurs when two branches have made different changes to the same lines of the same file, and Git cannot automatically decide which version to keep. Git stops the merge and marks the conflicting sections in the file with special markers. You must manually edit the file to resolve the conflict, then stage and commit the resolved file.

Conflict markers look like this in the file: <<<<<<< HEAD marks your current branch's changes, ======= is the separator, >>>>>>> feature/login marks the incoming branch's changes. Everything between these markers is the conflicting code you need to resolve manually.

Merge conflicts are a normal, expected part of collaborative development — not errors or failures. Learning to resolve them calmly and correctly is an essential professional skill. Most code editors (VS Code, IntelliJ) have built-in merge conflict resolution tools that show the conflicting sections visually with Accept/Reject buttons.

Conflict Example and Resolution
# What a conflict looks like in a file:
<<<<<<< HEAD (current branch - main)
  <h1>Welcome to our website</h1>
=======
  <h1>Welcome to the platform</h1>
>>>>>>> feature/new-hero (incoming branch)

# Resolution: Choose one, combine both, or rewrite:
  <h1>Welcome to our platform</h1>

# After resolving all conflicts:

git commit                   # Complete the merge

# Check merge status

5.5 Collaboration Strategies

Feature Branch Workflow is the most common strategy: every new feature, bug fix, or change gets its own branch. The main branch always contains only production-ready, stable code. Developers create a feature branch, do their work, push to GitHub, open a Pull Request, get code reviewed, and merge when approved.

GitFlow Workflow is a more structured strategy used in large projects with regular release cycles. It involves long-lived branches: main (production), develop (integration), feature/* (individual features), release/* (release preparation), and hotfix/* (urgent production fixes).

GitHub Flow is a simplified version of GitFlow, ideal for continuous deployment environments. It uses only main and feature branches. When a feature branch is merged to main, it is deployed immediately. This is the workflow used by GitHub itself.

10.1 Git Rebase

Git rebase is the process of moving or re-applying commits from one branch to another, creating a cleaner, more linear history. Unlike merge (which creates a merge commit preserving the fork history), rebase replays your commits on top of another branch as if you had started your work from the latest commit of that branch. This results in a perfectly linear commit history that is much easier to read and understand.

The interactive rebase (git rebase -i) is one of the most powerful Git tools available. It lets you rewrite your commit history before pushing: squash multiple commits into one, reorder commits, edit commit messages, split a commit, or drop unwanted commits. This is extremely useful for cleaning up messy working branches before submitting a Pull Request.

Golden Rule of Rebasing: Never rebase commits that have been pushed to a shared public branch. Rebase rewrites commit history (creates new commit hashes), which causes severe problems for teammates who have the old commits. Only rebase on your local feature branches before pushing, or on personal branches that nobody else uses.

Rebase Commands
# Basic rebase: replay feature branch on top of main
git switch feature/my-feature
git rebase main

# Interactive rebase (last 4 commits)
git rebase -i HEAD~4

# Interactive options:
# pick    = keep commit as is
# reword  = keep commit but edit message
# edit    = pause to amend commit
# squash  = merge into previous commit
# fixup   = like squash but discard message
# drop    = remove commit entirely

# Abort a rebase in progress
git rebase --abort

# Continue after resolving conflicts during rebase
git rebase --continue

10.2 Git Stash

Git stash is a temporary shelf for incomplete work. It saves your current working directory changes (including staged and unstaged modifications) and reverts to a clean working state, without making a commit. This is incredibly useful when you need to suddenly switch context — for example, you are midway through a feature and an urgent bug report comes in requiring you to immediately switch to the main branch. Instead of committing half-done work, you stash it, fix the bug, and then restore (pop) your stash to resume.

Stashes are stored in a stack — you can have multiple stashes. The most recent stash is always referenced as stash@{0}. Stashes are stored locally and do not transfer to GitHub when you push.

Stash Commands
# Stash current changes
git stash

# Stash with a descriptive message
git stash push -m "WIP: adding user profile form"

# List all stashes
git stash list
# stash@{0}: WIP: adding user profile form
# stash@{1}: On main: temp changes

# Apply most recent stash (keeps it in stash list)
git stash apply

# Apply and remove from stash list
git stash pop

# Apply a specific stash
git stash apply stash@{1}

# Delete a specific stash
git stash drop stash@{0}

# Clear all stashes
git stash clear

# View stash contents
git stash show -p stash@{0}

10.3 Git Cherry-Pick

Git cherry-pick applies the changes from a specific commit (or set of commits) to your current branch, without merging the entire source branch. It's like picking individual cherries from a tree rather than taking the whole branch. This is useful when you want to backport a specific bug fix from a development branch to a stable production branch, or when you want to bring over just one commit from a feature branch without merging everything else.

Cherry-Pick Commands
# Cherry-pick a single commit (use its hash)
git cherry-pick a3f72b8

# Cherry-pick multiple commits
git cherry-pick a3f72b8 b8c39d1 c2e45f7

# Cherry-pick a range of commits
git cherry-pick a3f72b8..c2e45f7

# Cherry-pick without committing (stage only)

# Abort cherry-pick on conflict

10.4 Semantic Versioning

Semantic Versioning (SemVer) is a standardized version numbering convention used by virtually all software projects. The format is MAJOR.MINOR.PATCH — for example, 2.5.3. Each number has a specific meaning:

  • MAJOR — Incremented for incompatible, breaking API changes. Going from 1.x.x to 2.0.0 means something that worked before will break.
  • MINOR — Incremented for new features that are backward-compatible. Old code still works, new functionality is added.
  • PATCH — Incremented for backward-compatible bug fixes only. No new features, no breaking changes.

Pre-release versions use suffixes: 1.0.0-alpha, 1.0.0-beta.1, 1.0.0-rc.2 (release candidate). Build metadata can be appended: 1.0.0+20240101. Understanding SemVer is essential for managing dependencies (package.json, requirements.txt) and for releasing your own software.

10.5 Git Hooks

Git hooks are scripts that Git automatically executes before or after specific events like commits, pushes, merges, or checkouts. They live in the .git/hooks/ directory and can be written in any scripting language (bash, Python, Node.js). Hooks are incredibly powerful for enforcing code quality standards, running automated tests before commits, linting code, enforcing commit message formats, and preventing accidental pushes to protected branches.

Common hooks: pre-commit (runs before commit — use to run linters/tests), commit-msg (validates commit message format), pre-push (runs before push — can run full test suite), post-merge (runs after merge — useful for running npm install if package.json changed). The popular tool Husky makes managing git hooks in JavaScript projects easy.

🎤 Interview Questions — Module 10

Q1: When would you use git stash vs git commit?
Use git stash when you have incomplete, work-in-progress changes that you need to temporarily set aside to switch to another task. Stash is for temporary, informal saves. Use git commit when your changes represent a logical, complete unit of work that you want to permanently record in the project history with a meaningful message.
Q2: What is the golden rule of git rebase?
Never rebase commits that have already been pushed to a public/shared branch that others have pulled. Rebase creates new commit hashes (rewrites history), which causes anyone who pulled the original commits to have a diverged history that is extremely difficult to reconcile. Only rebase on local branches or personal branches that no one else is using.

11.1 GitHub Actions — CI/CD Automation

GitHub Actions is GitHub's built-in automation and CI/CD (Continuous Integration/Continuous Deployment) platform. It allows you to automate virtually any workflow — running tests when code is pushed, building Docker images, deploying to cloud services, sending notifications, generating documentation, and much more. Actions are defined in YAML files stored in .github/workflows/ in your repository.

Key concepts: A Workflow is an automated process (a YAML file). It contains one or more Jobs. Each job runs on a Runner (a virtual machine — Ubuntu, Windows, or macOS). Jobs contain Steps — individual commands or pre-built Actions from the GitHub Marketplace. Workflows are triggered by Events: push, pull_request, schedule (cron), manual trigger (workflow_dispatch), etc.

CI/CD is a DevOps practice where code changes are automatically built, tested, and deployed. Continuous Integration (CI) means every commit triggers automated builds and tests, catching bugs immediately. Continuous Deployment (CD) means passing code is automatically deployed to staging or production environments.

.github/workflows/ci.yml — Example Workflow
name: CI Pipeline

# Trigger: run on every push to main and on all pull requests
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      # Step 1: Get the code
      - name: Checkout code
        uses: actions/checkout@v4

      # Step 2: Set up Node.js
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      # Step 3: Install dependencies
      - name: Install dependencies
        run: npm install

      # Step 4: Run linter
      - name: Run ESLint
        run: npm run lint

      # Step 5: Run tests
      - name: Run tests
        run: npm test

      # Step 6: Build the app
      - name: Build
        run: npm run build

13.1 How Recruiters Evaluate GitHub Profiles

Recruiters and hiring managers increasingly use GitHub as a primary evaluation tool for technical candidates. Understanding what they look for allows you to intentionally build a profile that makes a strong impression. Here is what experienced technical recruiters examine:

Contribution Activity: The green contribution graph on your profile page is immediately visible. Sudden "green washes" (bulk-committing hundreds of empty files) are easily spotted and viewed very negatively.

Code Quality in Repositories: Recruiters (and technical interviewers) may look at your actual code. Clean, well-commented, organized code with consistent style demonstrates technical maturity. Messy, uncommented spaghetti code is a red flag even for junior roles.

README Quality: Well-written READMEs with project descriptions, installation instructions, screenshots, and tech stacks show communication skills and attention to detail — both highly valued in professional environments.

Diversity of Projects: A portfolio that shows range — web apps, CLI tools, algorithms, API integrations, open-source contributions — demonstrates a broad and growing skill set. All similar "tutorial" projects suggest you haven't explored independently.

Collaboration Evidence: Having contributions to other repositories (PRs merged into external projects) is a significant positive signal. It proves you can work in unfamiliar codebases, communicate with maintainers, and follow external code standards.

Interview Questions with Detailed Answers

Comprehensive Q&A covering all topics — from basic concepts to advanced scenarios.

📘 Fundamentals

Q: What is the difference between git reset and git revert?
git reset moves the branch pointer backward to a previous commit, effectively removing commits from history. It has three modes: --soft (keeps changes staged), --mixed (keeps changes unstaged), --hard (discards all changes). It rewrites history and should only be used on local/private branches.

git revert creates a new commit that undoes the changes of a previous commit, without removing it from history. It is safe to use on shared branches because it does not rewrite history — it adds to it.
Q: What is HEAD in Git?
HEAD is a special pointer that indicates which branch or commit you are currently working on. In normal operation, HEAD points to a branch (e.g., "HEAD → main"). When you switch branches, HEAD moves to the new branch. When you checkout a specific commit directly (not a branch), you enter "detached HEAD" state — HEAD points directly to a commit rather than a branch. Any commits made in detached HEAD state are not on any branch and may be lost.
Q: What is the .git directory and what should you never do to it?
The .git directory is the heart of your Git repository — it contains all commit history, branch information, configuration, staged changes, and all Git metadata. It is created by git init. You should NEVER manually edit files inside .git, delete it (it would destroy your entire history), or copy it independently. Git commands manage .git correctly; manual edits can corrupt your repository irreversibly.

🌿 Branching & Merging

Q: What is a detached HEAD state and how do you recover from it?
Detached HEAD occurs when HEAD points to a specific commit instead of a branch. This happens when you checkout a commit hash, tag, or remote-tracking branch directly. In this state, any new commits are not attached to any branch and will be lost when you switch branches. To recover: 1) If you haven't made new commits, simply switch to any branch (git switch main). 2) If you made commits you want to keep, create a new branch at the current position: git switch -c new-branch-name. This saves your work onto a proper branch.
Q: Explain fast-forward vs three-way merge with an example.
Fast-forward merge happens when the target branch (main) has not diverged from the feature branch — main has no new commits since the branch was created. Git simply moves the main pointer forward to the feature branch's latest commit. No merge commit is created. The history appears linear.

Three-way merge happens when both branches have new commits since they diverged. Git uses three commits as reference points: the two branch tips and their common ancestor. It creates a new merge commit with two parents. This preserves the complete history of both development lines and is the most common merge in team environments.

🔄 Workflows & Collaboration

Q: What is a Pull Request and why is it important?
A Pull Request (PR) is a GitHub feature that formally proposes merging changes from one branch into another. It is NOT a Git concept — it is a GitHub collaboration tool. PRs are important because they: 1) Trigger code review by teammates before merging, 2) Display a diff of all proposed changes, 3) Allow inline comments on specific lines of code, 4) Trigger CI/CD checks to verify tests pass, 5) Maintain a permanent record of discussion and decisions about the change, 6) Can require specific approvals before merging. In professional environments, NO code goes directly to main — it always goes through a PR.
Q: What is the difference between GitHub Flow and GitFlow?
GitHub Flow is simpler: only two types of branches — main (always deployable) and feature branches. Any merged PR to main is immediately deployed. It is ideal for continuous deployment environments and small-to-medium teams.

GitFlow is more structured with multiple long-lived branches: main (production), develop (integration), feature/*, release/*, and hotfix/*. It was designed for projects with scheduled releases and strict versioning. GitFlow provides more control but requires more discipline and is more complex. Most modern teams use GitHub Flow for its simplicity.
Q: How do you handle a situation where you accidentally committed sensitive data (API key) to a public repository?
This is a serious incident. Steps: 1) IMMEDIATELY revoke/rotate the compromised credential in the service that issued it (AWS, GitHub, Google, etc.) — assume it's already been scraped by bots. 2) Remove it from history using git filter-branch or BFG Repo-Cleaner. 3) Force-push the cleaned history. 4) Contact GitHub Support if needed. 5) Add the file to .gitignore to prevent future accidents. 6) Use environment variables and tools like git-secrets or pre-commit hooks going forward. Remember: deleting the file and committing is NOT sufficient — the credential remains in Git history and GitHub cache.
